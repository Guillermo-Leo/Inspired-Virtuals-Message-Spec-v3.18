BMS Responses

BMS Response message

The Controller expects responses from the BMS to have a "message" root-element that follows the same format as messages sent to the BMS, with the exception of the xsi:type attribute, which is not required.

The MessageType attribute for message responses must be a string of either 'EventCard_Response', 'NoMoreBets_Response', ' Result_Response', 'Void_Response' or ‘LeagueUpdate_Response’ depending on what type of message is being responded to. The ControllerID and TransactionId attributes must be the same as the message the BMS is responding to.

Within the response message's message element there must be an event element with attributes:

  • Status – either 'ACK' or 'NACK'.
  • Details – a string for the client’s system to describe its status.
  • Action – either 'RETRY', 'VOID' or 'ERROR'. Only present if Status is 'NACK'

Unless the response is a Nack Retry, a Nack Error or a response to an Event Card message the following attributes are also required:

  • EventId – an integer that uniquely identifies the event.
  • EventGUID – a GUID that uniquely identifies the event.
  • EventType - an integer that uniquely identifies the event type.
  • EventTime - the time the event is due to start.

Controller reaction

The controller has a number of ways it can react to a BMS response:

  • Continue - The controller carries on with the event cycle
  • Retry - The controller sends the same message again.
  • Shutdown - The controller immediately shuts down and will require manual intervention to restart. This is the only time a controller will shutdown because of comms issues.
  • Remove Event - The controller removes the event without any further communication with the BMS

The table below shows how the controller reacts to each possible response for a given message

Ack NACK Error NACK Retry NACK Void No Response/Unknown Response
EventCard Continue Shutdown Retry Retry Retry
No More Bets Continue Shutdown Retry Remove Event Retry
Result Continue Shutdown Retry Continue Retry
Void Continue Shutdown Retry Remove Event Retry
LeagueUpdate Continue Shutdown Retry Continue Retry
GoalTimings Continue Shutdown Retry Continue Retry

If there is a delay that puts the controller behind schedule the controller will rapidly deal with events that are too late to be run. Depending on configuration it will either:

  • The controller will void all events that are in the past and send a void message to the BMS for each event
  • The controller will send a No More Bets message, result the event and then send the result message to the BMS.

The controller has a heartbeat file that is constantly updated while BMS comms is working correctly and stops being updated in the event of a BMS repeatedly not responding or sending NACK Retry messages. The heartbeat file can be monitored by Nagios or other similar software to alert support in case of a problem.

BMS requirements

Because the controller may not always receive any response sent by the BMS it is possible it will resend a message that the BMS has already seen and responded to. The BMS must be able to handle receiving duplicated messages.

The Integration Checklist contains a variety of tests that should be run on a BMS to ensure that the BMS responds correctly to all messages.
